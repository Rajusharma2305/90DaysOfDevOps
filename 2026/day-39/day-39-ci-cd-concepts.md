Task 1: The Problem

Imagine a team of 5 developers working on the same application.

Scenario Without CI/CD

Developer A writes Login feature.

Developer B writes Payment feature.

Developer C fixes bugs.

Developer D updates database code.

Developer E changes frontend UI.

Everyone pushes code and manually deploys to production.

What Can Go Wrong?
1. Code Conflicts

Two developers modify the same file.

Example:

# Developer A
tax = price * 0.18

# Developer B
tax = price * 0.20

When merging:

git merge main

Conflict occurs.

2. Bugs Reach Production

Developer pushes code without testing.

Example:

print(10/0)

Application crashes after deployment.

3. Manual Deployment Mistakes

Someone forgets a step.

Example:

git pull
docker build .
docker push image

Forgot:

docker restart app

New version never goes live.

4. Different Environments

Developer laptop:

Python 3.12

Production server:

Python 3.8

Application behaves differently.

What Does "It Works On My Machine" Mean?

Developer says:

"The application works perfectly on my laptop."

But when deployed:

Server Error
Application Failed

Why?

Different:

OS
Python version
Node version
Environment variables
Installed packages

Example:

Laptop:

node 22

Server:

node 18

Code works locally but fails in production.

This is one of the biggest reasons CI/CD exists.

How Many Times Can Teams Safely Deploy Manually?

Manual deployments are risky.

Typical team:

1-2 deployments/day

Maybe:

5 deployments/day

if highly disciplined.

With CI/CD:

10
50
100+
deployments/day

become possible.

Task 2: CI vs CD
1. Continuous Integration (CI)
Definition

Developers frequently merge code into a shared repository.

Every code push automatically:

Builds application
Runs tests
Checks quality
Goal

Catch problems early.

What CI Catches
Syntax errors
Failed tests
Broken builds
Merge issues
Example

Developer pushes code:

git push origin main

Pipeline automatically:

Build
↓
Run Tests
↓
Report Result

If tests fail:

❌ Build Failed

Code should not proceed.

Real World Example of CI

A React developer pushes code.

GitHub Actions:

Install dependencies
↓
Run npm test
↓
Run lint
↓
Success

Only then is code accepted.

2. Continuous Delivery
Definition

Code is always kept in a deployable state.

After CI passes:

Build
↓
Test
↓
Package
↓
Ready For Deployment

But deployment still requires approval.

Delivery Means

Application is ready to release anytime.

Human decides when.

Example

Pipeline:

Build
↓
Test
↓
Docker Image Created
↓
Stored in Registry

Manager clicks:

Deploy
Real World Example

Banking applications often use Continuous Delivery.

Reason:

Human approval required before production release.

3. Continuous Deployment
Definition

Every successful change is automatically deployed.

No manual approval.

Flow
Push
↓
Build
↓
Test
↓
Deploy

Everything automated.

Real World Example

Many SaaS companies use Continuous Deployment.

Example:

New bug fix pushed.

Within minutes:

Automatically live for users
CI vs Delivery vs Deployment
Continuous Integration
----------------------
Build + Test

Continuous Delivery
-------------------
Build + Test + Ready to Deploy

Continuous Deployment
---------------------
Build + Test + Automatically Deploy
Task 3: Pipeline Anatomy
1. Trigger

Starts the pipeline.

Examples:

on:
  push:

or

on:
  pull_request:
Example Flow

Developer:

git push

Trigger activates.

2. Stage

A major phase of pipeline.

Example:

Build Stage
Test Stage
Deploy Stage

Think of stages as chapters in a book.

3. Job

A group of related tasks.

Example:

jobs:
  build:

Build job may:

Install dependencies
Compile code
Create Docker image
4. Step

Smallest unit of work.

Example:

steps:
  - run: npm install

Another step:

- run: npm test

Each command is a step.

5. Runner

Machine executing jobs.

GitHub provides:

ubuntu-latest
windows-latest
macos-latest

Example:

runs-on: ubuntu-latest

Runner executes all commands.

6. Artifact

Output generated by pipeline.

Examples:

JAR file
ZIP file
Docker Image
Test Report

Artifact produced in one stage can be used later.

Example:

Build Stage
↓
app.jar
↓
Deploy Stage
